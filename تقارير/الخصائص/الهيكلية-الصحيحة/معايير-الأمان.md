# معايير الأمان

## 1. المصادقة

| السطح | الآلية | الملف |
|-------|--------|-------|
| لوحة التحكم | Supabase session + SystemUser | middleware · api-auth · layout |
| API إداري | `requirePermission()` | api-auth.ts |
| الكشك | `x-kiosk-key` header | kiosk-auth.ts |
| الصفحات SSR | `requirePagePermission()` | page-auth.ts |

### قواعد

- كل API route إداري **يجب** أن يبدأ بـ `requirePermission` أو `requireAuth`
- لا تعتمد على إخفاء الأزرار في UI فقط — الخادم هو الحكم النهائي
- `SystemUser.isActive = false` يمنع الوصول حتى مع جلسة Supabase صالحة

## 2. التعرف على الوجه

| الطبقة | الحماية |
|--------|---------|
| عميل | مطابقة + gap + 3 لقطات متتالية |
| خادم | `verifyFaceDescriptor` عند checkin/checkout |
| تسجيل | `findEmployeeByFaceDescriptor(..., "duplicate")` |

### ثغرة معروفة

الخادم يستخدم عتبة مسافة فقط **بدون** فحص gap — راجع `مخالفات-الممارسات/ديون-تقنية.md`.

---

## 3. الأسرار والبيئة

| المتغير | أين يُستخدم | آمن في العميل؟ |
|---------|-------------|----------------|
| `NEXT_PUBLIC_SUPABASE_URL` | عميل + خادم | ✅ نعم |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | عميل + خادم | ✅ (محدود بـ RLS — لا RLS حالياً على Prisma) |
| `DATABASE_URL` | خادم فقط | ❌ |
| `KIOSK_API_KEY` | يُمرَّر للعميل في الكشك | ⚠️ مرئي |
| `SUPABASE_SERVICE_ROLE_KEY` | سكربتات فقط | ❌ |
