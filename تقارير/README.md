# تقارير نظام الحضور والانصراف

مرجع تقني داخلي لصيانة النظام وإضافة ميزات جديدة بأمان واتساق.

**آخر تحديث:** يونيو 2026  
**المشروع:** `attendance-system/` — Next.js 14 · Prisma · Supabase · Human 1024-d  
**الإنتاج:** [attendance-zeta-flax.vercel.app](https://attendance-zeta-flax.vercel.app)  
**المستودع:** [murtadhaliwaa/Attendance](https://github.com/murtadhaliwaa/Attendance)

---

## هيكل هذا المجلد

```
تقارير/
├── README.md                          ← أنت هنا (فهرس عام)
├── 00-مرجع-النظام-الشامل.md          ← نقطة البداية: خريطة كاملة للنظام
└── الخصائص/
    ├── README.md                      ← فهرس الخصائص
    ├── تطويرات/                       ← توثيق كل ميزة وارتباطاتها
    ├── الهيكلية-الصحيحة/              ← كيف تُضاف ميزة جديدة بالطريقة الصحيحة
    └── مخالفات-الممارسات/             ← ما لا يتبع أفضل الممارسات + خطة الإصلاح
```

---

## متى تقرأ ماذا؟

| الهدف | الملف |
|--------|--------|
| فهم النظام بالكامل | `00-مرجع-النظام-الشامل.md` |
| إضافة ميزة جديدة | `الخصائص/الهيكلية-الصحيحة/دليل-إضافة-ميزة-جديدة.md` |
| معرفة أين يُعدَّل كود ميزة موجودة | `الخصائص/تطويرات/` |
| تجنب تكرار الأخطاء | `الخصائص/مخالفات-الممارسات/` |
| تتبع الارتباطات بين الملفات | `الخصائص/الهيكلية-الصحيحة/ارتباطات-الملفات.md` |

---

## قواعد استخدام هذا المرجع

1. **اقرأ قبل التعديل** — خاصة عند لمس المصادقة، الوجه، أو الكشك.
2. **حدّث التقرير** عند إضافة ميزة جديدة (ملف في `تطويرات/` + تحديث الارتباطات).
3. **لا تكرر المنطق** — راجع `مخالفات-الممارسات/تكرار-الكود.md` أولاً.
4. **الأمان أولاً** — أي API جديد يحتاج حارس صلاحيات أو مفتاح كشك.
